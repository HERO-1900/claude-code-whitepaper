读图说明：主流水线从左到右共 5 段——源码扫描 → AST 解析 → LLM 生成 → 模板引擎 → 输出写入，每一段颜色独立（青 / 紫 / 琥珀 / 翡翠 / 橙），对应源码中的 5 个独立包。下方灰色带是「缓存层」横切：Scanner 算出文件指纹后先问缓存；命中（87% 的场景）就跳过 Parser 与 LLM，直接复用上次的文档——这是 MagicDocs 能做到「每次改完代码几秒钟出新文档」的关键。右上角的玫红色方块是 `.magicdocs.json` 配置，通过 5 条虚线下发给每一个模块（路径白名单、使用的 Claude 模型、模板目录等）。右侧灰色面板是一份典型输出示例——`docs/auth/login.md`，包含用途 / 参数 / 返回 / 异常四段，底部带有「cached · 2ms」标签，说明这次走了缓存。 Reading guide: the main pipeline runs left-to-right through 5 stages — Scanner → Parser → LLM → Templater → Writer — each with its own color (cyan / violet / amber / emerald / orange) mapping to 5 separate packages in the source. The gray band below is the cross-cutting Cache Layer: after Scanner fingerprints a file it asks the cache first; on hit (87% of cases) it skips Parser & LLM entirely and reuses last output — this is why MagicDocs can emit fresh docs seconds after an edit. The rose block at top-right is `.magicdocs.json`, fanned out to all 5 modules via 5 dashed arrows (path whitelist, Claude model choice, template dir, etc.).

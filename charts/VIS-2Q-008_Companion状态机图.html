为什么需要 AWAITING_AUTH？ Why AWAITING_AUTH?
当 Companion 端主进程请求敏感工具（如 Bash、文件写入）时，默认行为不是直接执行，而是把一个带上下文的授权请求推送到手机 App。用户在手机上看到"允许/拒绝"二选一；这是 Companion 模式相对 CLI 的核心隔离点。 When the main process requests a sensitive tool (Bash, file write), the default is not direct execution but a contextual push to the mobile app. The user sees an Allow/Deny prompt on the phone. This is the core isolation boundary of Companion mode vs raw CLI.
30s 超时策略 30s timeout
任意非 IDLE 态在 30 秒内没有进一步事件，都会强制回落 IDLE，避免手机离线或用户分心造成僵死会话。LISTENING/PROCESSING/AWAITING_AUTH 共享同一超时时钟，由守护线程 tick。 Any non-IDLE state with no further events within 30s is forced back to IDLE, preventing stuck sessions when the phone is offline or the user is distracted. LISTENING/PROCESSING/AWAITING_AUTH share one watchdog timer.
ACTIVE 为何双边框？ Why double border on ACTIVE?
双边框在 UI 语义上代表"已越过授权门槛正在运行",是全局唯一能修改用户环境的态。设计上给予视觉最高权重,提醒读者：只有进到内圈才会产生副作用。 The double ring semantically marks "past the authorization gate and executing" — the only state that mutates the user's environment. It gets the heaviest visual weight to remind readers: side effects live only inside the inner ring.
Pentagon 布局的取舍 Pentagon layout tradeoff
5 个状态用五边形排列而非线性流，是为了让"任意态 → IDLE"这条超时回边在视觉上被 IDLE 居中辐射承接。代价是 PROCESSING → ACTIVE 这条对角线要横穿中心，用绿色且避开节点。 Five states are arranged in a pentagon (not a line) so timeout edges from any state can visually converge on IDLE at the top. The cost: the PROCESSING → ACTIVE diagonal has to cross the center — rendered green and routed around nodes.

Cross-Source Merge · CLI / Policy / Project / User / Default Across config sources · CLI / Policy / Project / User / Default
first-source-wins
示例 / ExampleExample
合并过程Merge Process 
// Source P1 — CLI (最高优先级highest priority)
{ "model": "claude-opus" }

// Source P2 — Policy (次高second highest)
{
  "model":        "claude-haiku",
  "allowedTools": []
}

// ↓ Result: P1 wins on "model"; P2 fills gap
{
  "model":        "claude-opus",  // ← P1 保留keep
  "allowedTools": []            // ← P2 补空fill gap
}
适用场景 / When It AppliesWhen It Applies
▸
CLI flag（--flag-name）优先于所有其他来源 CLI flags (--flag-name) take precedence over all other sources
▸
Policy 层覆盖 Enterprise → Project → Local 层 Policy layer overrides Enterprise → Project → Local layers
▸
User 级配置（~/.claude/settings.json）覆盖 Default User-level config (~/.claude/settings.json) overrides Default
▸
不同 source 的同名 key 冲突时，优先级高者胜出 When different sources conflict on the same key, the higher-priority one wins
⚠ GOTCHA 说明⚠ GOTCHA
低优先级来源永远无法覆盖高优先级来源中已定义的 key。 即使 Project 层写了 "model": "claude-haiku"， 用户的 CLI flag 也始终生效。这是安全设计，防止项目配置劫持用户意图。 A lower-priority source can never override a key already defined in a higher-priority source. Even if the Project layer sets "model": "claude-haiku", the user's CLI flag always takes effect. This is a safety design that prevents project config from hijacking user intent.

Same-Source Merge · CLAUDE.md 目录树继承链 Same-Source Merge · CLAUDE.md directory-tree inheritance chain
last-write-wins + deep merge
示例 / ExampleExample
合并过程Merge Process 
// Parent — ./CLAUDE.md
{
  "tools": ["Read", "Edit"]
}

// Child — ./subdir/CLAUDE.md
{
  "tools": ["Bash"],
  "rules": ["no-sudo"]
}

// ↓ Result: deep merge, arrays concatenated
{
  "tools": [
    "Read", "Edit",  // ← parent
    "Bash"               // ← child 追加append
  ],
  "rules": ["no-sudo"]   // ← child 独有unique
}
适用场景 / When It AppliesWhen It Applies
▸
子目录 CLAUDE.md 继承并扩展父目录的配置 Child CLAUDE.md inherits and extends parent-directory config
▸
数组（如 tools / rules）全部拼接，不去重 Arrays (e.g. tools / rules) are fully concatenated, without dedup
▸
对象字段递归深合并（nested objects deep-merged） Object fields are recursively deep-merged (nested objects deep-merged)
▸
同 source 内后定义的 scalar key 覆盖先定义（last-write-wins） Within the same source, later scalar keys override earlier ones (last-write-wins)
⚠ GOTCHA 说明⚠ GOTCHA
数组拼接是累积式的，不会去重。 若父目录和子目录都声明了同一工具，工具会被调用两次或重复列出。 若需覆盖而非追加，必须切换到不同的 source 层（跨源合并）。 Array concatenation is cumulative and does not deduplicate. If parent and child directories declare the same tool, that tool gets invoked twice or listed redundantly. To override instead of append, you must switch to a different source layer (cross-source merge).
⚡
关键区别 / Key DifferenceKey Difference
跨源合并用 first-source-wins—— 防止低优先级来源覆盖高优先级来源，最高优先级（CLI）始终胜出，保护用户意图不被项目配置劫持；   同源合并用 last-write-wins + 深合并—— 后代目录的 CLAUDE.md 继承祖先配置并叠加扩展，数组拼接、对象递归，让子目录获得更丰富能力而不是重写父级规则。 Cross-source merge uses first-source-wins—— it prevents lower-priority sources from overriding higher-priority ones, and the highest-priority source (CLI) always wins, protecting user intent from being hijacked by project config;   same-source merge uses last-write-wins + deep merge—— descendant CLAUDE.md files inherit ancestor config and stack extensions on top, arrays concatenate and objects recurse, so child directories gain richer capability instead of rewriting parent rules.
跨源合并 · Cross-SourceCross-Source
first-source-wins
高优先级来源的 key 不可被低优先级覆盖。CLI > Policy > Enterprise > Project > Local > User > Default。 Keys from a higher-priority source cannot be overridden by a lower one. CLI > Policy > Enterprise > Project > Local > User > Default.
同源合并 · Same-SourceSame-Source
deep-merge
数组拼接（concat），对象递归深合并（recursive merge），scalar 值后写覆盖先写（last-write-wins）。 Arrays concatenate (concat), objects recursively deep-merge (recursive merge), scalar values are overridden later-over-earlier (last-write-wins).
